# story-bible-qa

长篇小说 Story Bible 一致性检查器。

`story-bible-qa` 面向长篇小说作者、编辑和内容团队，目标不是替你“写小说”，而是帮助你在长周期创作中发现并修复角色设定、时间线、世界观、道具规则和章节事实之间的前后矛盾。

当前仓库已经包含一个可运行的前端原型和一套本地 TypeScript 连续性规则引擎，技术栈为 React + TypeScript + Vite。

## 项目介绍

长篇作品最难维护的部分，往往不是单章写作，而是跨几十万字、几百章之后的持续一致性：

- 角色年龄、关系、外貌和能力设定发生漂移
- 时间线前后打架，节日、年份、距离和行程无法闭合
- 世界观规则在不同章节中被悄悄改写
- 伏笔、道具、地点和事件状态出现“作者自己都忘了”的冲突
- 编辑审校依赖人工记忆，成本高且难以规模化

本项目希望把 Story Bible 从“静态资料库”推进到“可验证的事实基线”，再基于规则检查和 AI 辅助抽取，生成可追溯的问题报告。

## 为什么不做通用 AI 写作器

我们刻意不把产品定位成“通用 AI 写作器”，原因不是能力不足，而是产品边界要足够清晰：

- 通用写作器追求“生成更多文本”，而本项目追求“降低设定冲突与返工成本”
- 小说创作者对文风、节奏、人物声音有强主观控制需求，过度生成会削弱作者主导权
- 通用写作器最容易遭遇幻觉、跑题和风格污染；一致性检查器则更容易围绕“事实校验”建立信任
- “检查并举证”比“代写内容”更容易形成明确的工作流价值，也更适合团队协作与商业化
- 聚焦质量保证场景，产品差异化更强，开源与商业边界也更容易切分

一句话说，这不是一个替作者写小说的系统，而是一个帮助作者守住作品自洽性的 QA 系统。

## 目标用户

- 长篇网文作者：需要持续维护人物、设定和章节信息
- 小说编辑与工作室：需要批量检查项目质量，降低人工审校负担
- IP 孵化团队：需要在多人协作环境中统一世界观、角色卡和剧情资产
- 重度设定型创作者：尤其是奇幻、科幻、悬疑、群像与系列作品

## 核心能力设想

- Story Bible 导入：角色卡、世界观设定、地点、势力、时间规则、物品规则
- 章节事实抽取：从正文中提取角色状态、关系变化、事件、道具与时间地点线索
- 一致性规则检查：发现显性冲突、隐性冲突和缺失证据
- 问题报告生成：为每条问题附上原文证据、冲突说明和建议处理方向
- 修订回归检查：在章节重写或补丁修改后再次扫描，确认冲突是否关闭

## 商业可行性

这个方向的可行性，来自于它解决的是一个高频、真实且会随着作品规模放大的问题：

- 字数越长、连载越久，维护成本越高，需求天然增强
- “一致性错误”会直接影响读者口碑、订阅留存和编辑效率，价值足够具体
- 检查结果可被展示、讨论、关闭和复查，天然适合工作流产品
- 对个人作者可以做提效工具，对团队可以做协作与审校系统，客单价有上升空间
- 先做窄场景工具更容易建立口碑，再向上承接更完整的创作基础设施

更详细的分析见 [docs/market-analysis.md](./docs/market-analysis.md)。

## 开源边界

建议把“可信、可本地运行、可审计”的基础能力开源，把“协作效率、托管服务、规模化能力”留作未来商业层。

适合开源的部分：

- Story Bible 数据结构与导入导出格式
- 本地规则引擎与基础一致性检查
- 可追溯的问题报告格式
- 单机版或 CLI/本地 Web 版扫描流程
- 样例数据、评测集和开发者扩展接口

适合未来商业化的部分：

- 团队工作区、权限、评论流与任务状态
- 托管式长文本索引、项目资产管理和版本对比
- 更强的实体抽取、冲突聚合和批量项目分析
- 编辑后台、项目看板、统计报表和质量趋势追踪
- 私有部署、定制规则库、审校流程接入和 API 服务


## 使用说明

### 本地开发

```bash
npm install
npm run dev
```

默认会启动 Vite 开发环境。

### 构建与检查

```bash
npm run build
npm run lint
```

### 计划中的典型使用流程

1. 导入或录入 Story Bible 资料
2. 导入章节正文或项目文本
3. 运行一致性检查
4. 查看带证据的问题报告
5. 修订正文或设定后再次扫描
